Executive View

Ibtasim is one product with two money paths. First: standard project donations. Second: orphan sponsorship through kafala. The app combines a public donor website and an admin back office, with Convex acting as backend logic layer, schema layer, runtime storage, and part of the settings surface.

What the product isA charity platform for campaigns, sponsorship, donor journeys, and internal operations.
Why logic feels hardBehavior is spread across pages, Convex modules, callback handlers, and operational settings.
Why design feels unresolvedThe design system is documented better than it is enforced in runtime code.

      Risk Register

      Severity is ordered by real damage potential, not by code neatness. Security boundary issues come first, then money and state integrity, then frontend/runtime quality, then maintainability.

      P0 Security

      Backend authorization is weaker than the UI suggests

      • Client-side admin route checks are not a security boundary.
      • Operational modules expose sensitive reads or privileged writes too broadly.
      • Admin bootstrap and invitation behavior is especially sensitive.
      P1 Data Integrity

      Money units and transaction states are not normalized enough

      • Amount handling is inconsistent between UI creation, public display, and reporting assumptions.
      • Verification and callback paths look weakly idempotent.
      • Kafala and standard donations duplicate transaction concepts instead of sharing one hardened model.
      P2 Runtime Quality

      Frontend structure is buildable, not yet clean

      • Lint fails and points to structural/runtime issues.
      • Bad localStorage values can break startup behavior.
      • Placeholder links, partial accessibility wiring, and mixed patterns reduce confidence.
      P3 Maintainability

      The design system is documented better than it is enforced

      • Tokens and layout docs exist.
      • Pages still bypass them with heavy inline styling and duplicated visual logic.
      • New pages can drift fast because the component layer is not strict enough.

      What works now

      • The public product has a recognizable charity identity.
      • Teal for public donation and warm gold for kafala is a sensible split.
      • The admin interface clearly reads as an operations dashboard.
      • Cards, CTAs, and sectioned content are visually understandable.

      What breaks confidence

      • Inline style volume makes the product feel page-crafted rather than system-driven.
      • English labels and Arabic body content are mixed unevenly.
      • Spacing, badge logic, and component detail differ across page families.
      • Interactive states are not standardized strongly enough.

      What the design is not yet

      • It is not yet a strict, scalable design system.
      • It is not yet a complete multilingual product experience.
      • It is not yet one unified premium donor journey.
      • It is not yet safe from visual drift as more pages are added.

      UX Writing Audit Across Arabic, French, and English

      The language model exists in code, but the product writing system is not mature enough yet. The main issue is not only translation coverage. It is tone, completeness, vocabulary consistency, and interface hierarchy.

      Arabic

      Primary language

      • Strongest coverage in navigation, CTAs, and public-facing content.
      • Still mixed with English overlines and labels in important views.
      • Tone is often warm and charitable, but not fully standardized across product surfaces.
      • RTL support exists, but several components hardcode layout behavior instead of using a unified system.

      French

      Partial support

      • Core dictionary entries exist, but full page-level localization is incomplete.
      • There is a gap between translated UI tokens and a complete French donor journey.
      • Trust and payment messaging need native-quality copy, not only direct equivalents.
      • Today French behaves like a UI mode, not a finished product voice.

      English

      Partial support

      • English appears in labels and structure more than in full donor content.
      • Some pages mix English section labels with Arabic content, which weakens tone consistency.
      • English is serviceable as a fallback interface layer, not yet as a polished donor journey.
      • A complete English pass should include trust, payment instructions, empty states, and support text.

      Desktop and Mobile Audit

      The codebase intends to support both desktop and mobile. The problem is that responsive behavior is split across inline styles, Tailwind utilities, and page-specific layout choices. That makes the result acceptable, but not reliably excellent.

      Desktop

      • Desktop pages have enough space to feel credible and operational.
      • Admin pages fit desktop usage better because density is expected there.
      • Public pages feel more visually stable on desktop than mobile.
      • The desktop experience is the strongest current version of the product.

      Mobile

      • Mobile containers and sticky behavior are documented and partially implemented.
      • The app can function on mobile, but not every page feels designed mobile-first.
      • Form-heavy and admin-heavy screens need clearer mobile hierarchy.
      • Component size and spacing are not consistently token-driven.

      Main responsive problems

      • Too many layout decisions live inside page files.
      • Some desktop-rich compositions do not collapse into a strong mobile grammar.
      • Navigation and action density differ by page family.
      • Responsive behavior should be part of component contracts, not improvisation.

      How to Fix Your Two Main Problems

      You said the hardest part is understanding the full logic and then reaching a design you are happy with. Those are two different workstreams and they should be solved in this order.

      Step 1: Logic pack

      • Use this HTML as the system map.
      • Validate each flow against code while keeping the visual map open.
      • Normalize statuses, payment states, and money units on paper first.
      • Do not redesign flows you still do not fully understand.

      Step 2: Design recovery

      • Choose one visual direction for the whole product.
      • Lock tokens: color, type, spacing, radius, shadow, and motion.
      • Design six core primitives before redesigning full pages.
      • Redesign one flagship page first, then systematize.

      Step 3: Writing and responsive pass

      • Create a content style guide for Arabic, French, and English.
      • Audit every CTA, empty state, success state, error state, and helper text.
      • Make mobile behavior part of component contracts, not page-level improvisation.
      • Only then scale design across the rest of the product.
